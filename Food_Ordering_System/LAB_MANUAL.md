## INTRODUCTION & GOALS

This project walks you through building a fully working food delivery web portal called **QuickBite**. You will design its database, generate backend code using a framework, style it for all device sizes, and wire everything together using REST principles.

### What You'll Learn
- How MVC frameworks separate data, logic, and display layers
- How to generate models and basic CRUD screens using framework tools
- How to make a web interface responsive using Bootstrap
- How to choose the right HTTP verb for each type of operation
- What it means for an API to be stateless and idempotent
- How to structure clean, RESTful URI paths for your resources

### Framework Choices
Pick exactly **one** of:
- **Laravel (PHP)** — Great for beginners, outstanding documentation
- **Django (Python)** — Built-in admin panel, powerful ORM
- **ASP.NET Core (C#)** — Enterprise-ready with strong tooling

## APPLICATION OVERVIEW

QuickBite is a three-layer web application: customers browse a restaurant's digital menu, build and submit orders, and track their delivery status. Managers have admin access to add, edit, or remove items.

### Functional Requirements

| Capability | Details |
|-----------|---------|
| Dish Catalog | Managers can add/edit/remove dishes with name, description, price, and type |
| Customer Orders | Customers browse, place, and track food orders |
| Multi-Device Support | All screens work equally well on phones, tablets, and desktops |
| REST Architecture | All data operations go through properly structured REST endpoints |
| Persistent Storage | A relational database (SQLite, MySQL, or PostgreSQL) holds all data |

### Database Entities

Your project must define these tables at minimum:

- **Dish**: id, title, description, dish_type, unit_price, is_available
- **CustomerOrder**: id, buyer_name, placed_at, order_status, amount_due
- **OrderLine**: id, order_id (FK), dish_id (FK), qty, locked_price
- **Optional — Buyer**: id, full_name, email_address, contact_number, delivery_address

## STAGE 3: HTTP Methods & REST Architecture
**Timeframe: 1.5 weeks | Weight: 25% of total**

### Goals for This Stage
- Assign the correct HTTP verb to every action in your app
- Understand what stateless communication means and how to achieve it
- Classify every endpoint as idempotent or non-idempotent
- Write complete API documentation

### Core REST Concepts

#### Stateless Communication
Every HTTP request your app sends must be fully self-sufficient. The server should not remember anything about previous interactions. If a user is logged in, their credentials or session token must travel with every request—not be stored server-side.

This improves scalability: any server in a cluster can handle any request because there's no session pinned to a specific server.

**Concrete Example:**
```
Client → Server:
PUT /customer-orders/7 HTTP/1.1
Authorization: Bearer <jwt_token_here>
Content-Type: application/json
{ "order_status": "preparing" }

Server Response → Client:
HTTP/1.1 200 OK
{ "id": 7, "order_status": "preparing", ... }
```
The token travels with the request — the server never stored "who is logged in."

#### HTTP Verb Reference

| Verb | When to Use | Notes |
|------|------------|-------|
| GET | Fetch data | Never modifies anything. Safe + idempotent. |
| POST | Create a new record | NOT idempotent. Each call may create a new row. |
| PUT | Replace a whole record | Idempotent — same result every time. |
| PATCH | Update one or two fields | Partially modifies a record. |
| DELETE | Remove a record | Idempotent — deleting twice has the same end result. |

#### Idempotency Explained
An endpoint is **idempotent** if calling it five times produces the same result as calling it once, with no extra side effects.

**Idempotent in QuickBite:**
- `GET /dishes` — always returns the same list
- `PUT /dishes/3` — always sets that dish to the same state
- `DELETE /customer-orders/9` — the order is gone, whether you call once or twice

**Not idempotent in QuickBite:**
- `POST /customer-orders` — creates a brand-new order each time
- `POST /customer-orders/9/lines` — adds another copy of the same dish every time

### HTTP Method Mapping for QuickBite

| Action | Verb | Endpoint | Idempotent? |
|--------|------|----------|------------|
| Browse all dishes | GET | /dishes | Yes |
| View one dish | GET | /dishes/:id | Yes |
| Add a new dish | POST | /dishes | No |
| Edit a dish | PUT | /dishes/:id | Yes |
| Remove a dish | DELETE | /dishes/:id | Yes |
| Browse all orders | GET | /customer-orders | Yes |
| View one order | GET | /customer-orders/:id | Yes |
| Place a new order | POST | /customer-orders | No |
| Update order status | PUT | /customer-orders/:id | Yes |
| Cancel an order | DELETE | /customer-orders/:id | Yes |

### Framework Implementation Notes

**Laravel** — use `Route::resource()` in `routes/web.php` to map all seven RESTful methods automatically.

**Django** — use class-based views (`ListView`, `CreateView`, etc.) or Django REST Framework `ViewSet` with `@action` decorators.

**ASP.NET Core** — decorate controller actions with `[HttpGet]`, `[HttpPost]`, `[HttpPut]`, `[HttpDelete]`.

For PUT and DELETE from HTML forms, use a hidden input:
```html
<input type="hidden" name="_method" value="PUT">
```

### Documentation to Write: `REST_API_DOCUMENTATION.md`

This document must contain three sections:

1. **Statelessness** — Explain in your own words how your app is stateless. Show an example HTTP request that includes a token and explain why no server-side session is stored.

2. **Full Endpoint Reference** — For every endpoint: method, URI, what it does, request parameters or body, example curl command, and both success and error responses.

3. **Idempotency Table** — List all endpoints. Label each IDEMPOTENT or NOT IDEMPOTENT. Explain the reasoning.

## STAGE 4: RESTful URI Structure
**Timeframe: 1 week | Weight: 20% of total**

### Goals for This Stage
- Design clean, resource-oriented URL paths
- Use plural nouns (never verbs) in all URIs
- Represent resource hierarchies in the path structure
- Finalize and document all routes

### URI Rules (Strictly Enforced)

#### Rule A: Plural Nouns Only
Every URI must refer to a collection using a plural noun.

| Correct ✓ | Incorrect ✗ |
|-----------|------------|
| `/dishes` | `/dish` or `/getDish` |
| `/customer-orders` | `/order` or `/createOrder` |
| `/buyers` | `/buyer` or `/fetchCustomer` |

#### Rule B: No Verbs in Paths
The HTTP method defines the *action*. The URI defines the *resource*. Never mix them.

| Correct ✓ | Incorrect ✗ |
|-----------|------------|
| `POST /customer-orders` | `/submitOrder` |
| `DELETE /dishes/4` | `/removeItem/4` |
| `GET /dishes` | `/listAllDishes` |

#### Rule C: Show Relationships via Nesting
When one resource belongs to another, reflect that in the path:

```
GET /customer-orders/3/lines          → All line items in order 3
GET /customer-orders/3/lines/11       → One specific line item
GET /dishes/7/reviews                 → All reviews for dish 7
GET /buyers/2/customer-orders         → All orders from buyer 2
GET /dish-types/4/dishes              → All dishes in category 4
```

### Complete URI Table for QuickBite

| Method | URI | Purpose |
|--------|-----|---------|
| GET | /dishes | Get all dishes |
| GET | /dishes/:id | Get one dish |
| POST | /dishes | Create a dish |
| PUT | /dishes/:id | Update a dish |
| DELETE | /dishes/:id | Remove a dish |
| GET | /dish-types | Get all types |
| GET | /dish-types/:id/dishes | Dishes in a type |
| GET | /customer-orders | Get all orders |
| GET | /customer-orders/:id | Get one order |
| POST | /customer-orders | Submit an order |
| PUT | /customer-orders/:id | Update an order |
| DELETE | /customer-orders/:id | Cancel an order |
| GET | /customer-orders/:id/lines | All lines in an order |
| POST | /customer-orders/:id/lines | Add a line to an order |
| DELETE | /customer-orders/:id/lines/:lineId | Remove a line from order |
| GET | /buyers | List all buyers |
| GET | /buyers/:id/customer-orders | Orders by a buyer |

### Best Practices

- **Use lowercase:** `/dish-types` not `/DishTypes`
- **Hyphens between words:** `/customer-orders` not `/customerorders`
- **Limit nesting depth:** Generally no more than 2–3 levels deep
- **Filtering via query strings:** `/customer-orders?status=preparing&limit=5`

1. HTTP method + URI
2. Brief description
3. Whether it uses plural nouns (verify each one)
4. Whether it contains any verbs (should be none)
5. A short example request/response
